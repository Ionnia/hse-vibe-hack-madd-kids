================================================================================
                     AI TUTOR BOT — КАК РАБОТАЮТ АГЕНТЫ
================================================================================

В системе нет агентов в классическом смысле (LangChain Agents, ReAct-петель
с инструментами). Вместо этого каждый «агент» — это автономный Celery-воркер,
который выполняет одну специализированную задачу: получает входные данные,
вызывает LLM или внешний сервис, сохраняет результат в БД и завершается.

Агенты общаются через очередь Redis и общую БД PostgreSQL.
Они не знают друг о друге напрямую — каждый реагирует на своё событие.


────────────────────────────────────────────────────────────────────────────────
АГЕНТ 1 — INGESTION AGENT (агент загрузки)
────────────────────────────────────────────────────────────────────────────────

Файл: app/tasks/material_ingestion_tasks.py
      app/services/material_ingestion_pipeline.py

Когда запускается:
  Сразу после того как пользователь отправил текст, фото или аудио в бот.
  Бот вызывает ingest_material.delay(material_id).

Что делает:
  1. Загружает материал из БД по material_id
  2. Если фото — вызывает OCR адаптер (OpenAIOCR → GPT-4o-mini Vision)
     Если аудио — вызывает транскрибатор (LocalWhisperTranscription)
     Если текст — берёт как есть
  3. Нормализует текст (убирает лишние пробелы, переносы)
  4. Разбивает на чанки ~1000 символов (ChunkingService)
  5. Сохраняет чанки в topic_chunks
  6. Обновляет статус материала: uploaded → processing → готов к извлечению

После завершения:
  Ставит в очередь следующего агента — run_topic_pipeline.delay(material_id)

Схема:
  Telegram → бот → [Redis] → Ingestion Agent → [PostgreSQL] → [Redis] →


────────────────────────────────────────────────────────────────────────────────
АГЕНТ 2 — TOPIC EXTRACTION AGENT (агент извлечения тем)
────────────────────────────────────────────────────────────────────────────────

Файл: app/tasks/topic_extraction_tasks.py
      app/services/topic_pipeline_service.py
      app/services/topic_extraction_service.py

Когда запускается:
  После завершения Ingestion Agent — получает material_id из очереди.

Что делает:
  1. Загружает нормализованный текст материала из БД
  2. Отправляет первые 3000 символов в LLM с промптом:
     "Извлеки ключевые учебные темы. Верни JSON: [{name, text, subject}]"
  3. LLM (gpt-4o-mini) возвращает список тем на русском языке
  4. Каждая тема сохраняется в таблицу topics
  5. Статус материала → "ready"

После завершения:
  Для каждой новой темы ставит в очередь: enrich_topic_task.delay(topic_id)

Пример LLM-запроса:
  Промпт → "Текст: [нормализованный материал]... Верни только валидный JSON."
  Ответ  → {"topics": [{"name": "Закон Ома", "text": "...", "subject": "physics"}]}

Схема:
  → Topic Extraction Agent → LLM API → [PostgreSQL] → [Redis] →


────────────────────────────────────────────────────────────────────────────────
АГЕНТ 3 — ENRICHMENT AGENT (агент обогащения тем)
────────────────────────────────────────────────────────────────────────────────

Файл: app/tasks/topic_extraction_tasks.py → enrich_topic_task
      app/services/topic_enrichment_service.py
      app/adapters/web_search/duckduckgo.py

Когда запускается:
  Автоматически — сразу после создания каждой темы (цепочка от агента 2).
  Периодически — Celery Beat раз в час запускает enrich_pending_topics,
  который находит темы с enriched_at IS NULL и ставит их в очередь.
  Вручную — пользователь нажимает кнопку «🔍 Обогатить темы» в /topics.

Что делает (шаг за шагом):

  Шаг 1 — Проверка
    Загружает тему из БД. Если enriched_at уже заполнено — пропускает.
    (Защита от двойного обогащения.)

  Шаг 2 — Поиск в интернете
    Формирует поисковый запрос: "{topic.name} {topic.subject}"
    Пример: "Закон Ома physics"

    DuckDuckGoWebSearch запускает синхронный поиск в отдельном потоке
    (через asyncio executor, чтобы не блокировать event loop):

      with DDGS() as ddgs:
          results = ddgs.text(query, max_results=5)

    Получает список SearchResult: {title, url, snippet}

  Шаг 3 — Синтез через LLM
    Передаёт в OpenAILLM.enrich_topic():
      - Оригинальное название и текст темы
      - 5 сниппетов из поиска

    Промпт (упрощённо):
      "Вот текст темы: [оригинал]
       Вот что нашли в интернете: [5 сниппетов с URL]
       Дополни текст новыми фактами. Не противоречь оригиналу.
       Отвечай на русском, максимум 500 слов."

    LLM возвращает обогащённый текст.

  Шаг 4 — Сохранение
    Обновляет запись в БД:
      topic.text             = обогащённый текст
      topic.enriched_at      = текущее время UTC
      topic.enrichment_sources = [url1, url2, ...]  (JSON-массив)

  При ошибке:
    Задача повторяется до 3 раз с паузой 60 секунд (Celery retry).

Что видит пользователь:
  В /topics рядом с темой появляется ✅ если обогащение прошло.
  Следующие задания /study генерируются на основе обогащённого текста —
  вопросы становятся глубже, охватывают больше аспектов темы.

Полная схема агента обогащения:

  [Redis queue]
       ↓
  enrich_topic_task(topic_id)
       ↓
  TopicEnrichmentService.enrich(topic_id)
       ↓ ← проверка enriched_at
  DuckDuckGoWebSearch.search("{name} {subject}")
       ↓ ← asyncio executor (не блокирует)
  DDGS().text() → 5 SearchResult
       ↓
  OpenAILLM.enrich_topic(name, original_text, search_results)
       ↓ ← промпт + GPT-4o-mini
  enriched_text (str)
       ↓
  TopicRepository.update_enrichment()
       ↓
  [PostgreSQL] topic.text обновлён, enriched_at проставлен


────────────────────────────────────────────────────────────────────────────────
АГЕНТ 4 — TUTOR AGENT (агент генерации заданий)
────────────────────────────────────────────────────────────────────────────────

Файл: app/services/tutor_task_service.py
      app/bot/handlers/study.py

Когда запускается:
  Синхронно при команде /study — не через Celery, а прямо в боте,
  потому что пользователь ждёт ответа.

Что делает:
  1. Определяет уровень пользователя по теме (unknown/weak/learning/good/mastered)
  2. Загружает последние 10 вопросов по этой теме (история)
  3. Передаёт в LLM:
     - Текст темы (уже обогащённый, если агент 3 отработал)
     - Уровень сложности (1–5)
     - Список предыдущих вопросов (чтобы не повторяться)
  4. LLM генерирует новый вопрос + эталонный ответ + подсказки
  5. Задание сохраняется в tutor_tasks
  6. Бот отправляет вопрос пользователю

Пример LLM-промпта:
  "Сгенерируй учебное задание. Уровень: weak (сложность 2/5)
   Тема: Закон Ома. Содержание: [обогащённый текст]
   Уже заданные вопросы: [список 10 предыдущих]
   Верни JSON: {type, question, answer, difficulty, explanation, hints}"


────────────────────────────────────────────────────────────────────────────────
АГЕНТ 5 — REVIEW AGENT (агент проверки ответов)
────────────────────────────────────────────────────────────────────────────────

Файл: app/services/review_service.py
      app/bot/handlers/review.py

Когда запускается:
  Синхронно — когда пользователь отправил ответ на задание.

Что делает:
  1. Загружает задание (вопрос + эталонный ответ) из БД
  2. Передаёт в LLM.evaluate_answer():
     "Вопрос: [вопрос]
      Правильный ответ: [эталон]
      Ответ студента: [то что написал пользователь]
      Верни JSON: {verdict, score, feedback}"
  3. LLM возвращает: verdict (correct/partial/incorrect), score 0.0–1.0, фидбэк
  4. Записывает ReviewLog в БД
  5. Обновляет UserTopicProgress (пересчёт level)
  6. Обновляет RepetitionState (SM-2 алгоритм):
     - correct: интервал умножается на ease_factor (~2.5)
     - partial: интервал не меняется
     - incorrect: сброс к 1 дню, ease_factor -= 0.2
  7. Бот сообщает результат + следующую дату повторения


────────────────────────────────────────────────────────────────────────────────
АГЕНТ 6 — REPETITION AGENT (агент повторений)
────────────────────────────────────────────────────────────────────────────────

Файл: app/tasks/topic_extraction_tasks.py → send_due_repetitions

Когда запускается:
  Celery Beat — автоматически каждые 5 минут.

Что делает:
  1. Ищет в repetition_states все записи где next_review_at <= now()
  2. Логирует количество просроченных повторений
  3. (Готово к расширению: отправка Telegram-уведомлений пользователям)

Это фоновый сторож — он следит чтобы пользователи не забывали
повторять пройденные темы в нужное время.


────────────────────────────────────────────────────────────────────────────────
АГЕНТ 7 — Q&A AGENT (агент вопрос-ответ)
────────────────────────────────────────────────────────────────────────────────

Файл: app/bot/handlers/ask.py
      app/adapters/llm/openai_llm.py → answer_question()

Когда запускается:
  Синхронно при команде /ask — пользователь выбирает тему и задаёт вопрос.

Что делает:
  1. Показывает клавиатуру со всеми темами пользователя
  2. Пользователь выбирает тему (FSM state: choosing_topic)
  3. Пользователь пишет произвольный вопрос (FSM state: waiting_for_question)
  4. LLM.answer_question() отвечает:
     "Ты учебный помощник. Отвечай строго на основе материала темы.
      Тема: [название]. Материал: [текст темы, уже обогащённый].
      Вопрос студента: [вопрос]
      Если ответа нет в материале — честно скажи об этом."
  5. Бот отправляет ответ
  6. Остаётся в состоянии waiting_for_question — можно задать ещё вопрос

Отличие от /study:
  /study — агент сам придумывает вопрос и проверяет ответ пользователя
  /ask   — пользователь сам задаёт вопрос и получает объяснение от агента


────────────────────────────────────────────────────────────────────────────────
КАК АГЕНТЫ СВЯЗАНЫ МЕЖДУ СОБОЙ
────────────────────────────────────────────────────────────────────────────────

Пользователь загружает материал:

  Бот → [Redis]
    → Ingestion Agent (OCR/Whisper + нормализация + чанки)
    → [Redis]
    → Topic Extraction Agent (LLM → темы)
    → [Redis] × N тем
    → Enrichment Agent × N (DDG + LLM → обогащение)
    → [PostgreSQL] темы обогащены

Пользователь учится:

  /study → Tutor Agent (LLM → новый вопрос из обогащённой темы)
         → пользователь отвечает
         → Review Agent (LLM → оценка + SM-2 обновление)

  /ask   → Q&A Agent (LLM → ответ на произвольный вопрос)

Фон (Celery Beat):

  каждые 5 минут  → Repetition Agent (мониторинг просроченных тем)
  каждый час      → Enrichment Agent (подбирает необогащённые темы)


────────────────────────────────────────────────────────────────────────────────
ПРИНЦИПЫ АРХИТЕКТУРЫ АГЕНТОВ
────────────────────────────────────────────────────────────────────────────────

1. Один агент — одна ответственность
   Каждый агент делает ровно одно: OCR, или извлечение тем, или обогащение.
   Это упрощает отладку, повторные запуски и масштабирование.

2. Агенты не зависят друг от друга напрямую
   Связь только через Redis (очередь) и PostgreSQL (данные).
   Если один упал — другие продолжают работать.

3. Идемпотентность
   Enrichment Agent проверяет enriched_at перед работой — повторный запуск
   безопасен. Topic Extraction Agent не перезаписывает существующие темы.

4. Retry при ошибках
   Celery задачи с max_retries=3, countdown=60 секунд.
   LLM-ошибки (rate limit, timeout) не роняют всю систему.

5. NullPool для asyncpg в воркерах
   Celery форкает процессы. asyncpg не передаётся через fork.
   Каждая Celery-задача создаёт свой engine с NullPool и закрывает его
   после выполнения — никаких утечек соединений.

6. Обогащённый контекст улучшает все последующие агенты
   Tutor Agent и Q&A Agent читают topic.text — если Enrichment Agent
   уже отработал, они получают более богатый контекст и генерируют
   лучшие вопросы и ответы.
